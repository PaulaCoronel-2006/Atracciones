​Requerimientos Funcionales (RF)
Requerimientos Funcionales de Proceso
El sistema enviará un correo electrónico automatizado cuando se registre: una reserva confirmada, una cancelación por parte del usuario, el envío del voucher con QR y el registro del pago exitoso.
Se permitirá iniciar una reserva con datos básicos. Sin embargo, para emitir el voucher, los datos del "Viajero Principal" deben estar completos y validados.
Para atracciones que no son de "confirmación instantánea", la solicitud pasará a un estado de "Pendiente de Operador" hasta que el proveedor local apruebe el cupo, cambiando entonces el estado a "Confirmada".
A cada reserva se le asignará un Localizador Único Alfanumérico, que será el identificador en los procesos de check-in, auditoría y liquidación con el proveedor.
La facturación a proveedores se realizará en lotes semanales mediante una pantalla de "Reservas Ejecutadas" que mostrará sólo aquellas cuyo servicio ya fue prestado.
Al ingresar una cantidad de personas, el sistema calculará el total basándose en las reglas de edad de la atracción.
Al confirmar una reserva, el sistema restará automáticamente el número de tickets del inventario disponible para ese time slot y fecha específica.
Requerimientos Funcionales de Interfaz Gráfica
El campo de precio de la atracción aceptará únicamente valores numéricos con dos decimales, formateados según la moneda seleccionada.
El campo Fecha de Actividad aceptará únicamente fechas iguales o posteriores al día de hoy, considerando la zona horaria local de la atracción.
El campo País será una lista de preselección. Al elegir un país, el campo Provincia/Estado se filtra automáticamente.
El campo de búsqueda  mostrará sugerencias basadas en tres niveles: Ciudad, Categoría o Nombre específico de la atracción.
La galería de la atracción deberá mostrar miniaturas de fotos y soportar la reproducción de video en formato responsivo para móviles.
La interfaz mostrará un marcador geolocalizado con las coordenadas exactas de la atracción o el punto de encuentro.
Requerimientos Funcionales Legales y Regulatorios
Todas las modificaciones en precios de atracciones o cambios de estado de reservas (de Confirmada a Cancelada) deben registrarse en una tabla de auditoría con usuario, fecha, hora y valor anterior.
El sistema emitirá comprobantes electrónicos de venta siguiendo los formatos legales del país de operación del proveedor (ej: facturación electrónica del SRI en Ecuador o AFIP en Argentina).
El sistema debe permitir al usuario ejercer su derecho de "olvido" o eliminación de cuenta, anonimizando sus datos personales según el reglamento GDPR o leyes locales.
El sistema generará reportes mensuales de "Tasa Turística" para las autoridades municipales en ciudades donde este impuesto sea obligatorio.
Requerimientos de Seguridad
Solo usuarios autenticados podrán acceder al historial de reservas. El acceso administrativo para proveedores requerirá autenticación de dos pasos.
El sistema enviará una alerta al administrador cuando se detecten 3 intentos fallidos de pago con tarjetas de crédito distintas para una misma sesión.
Toda comunicación entre el cliente y el servidor se realizará bajo protocolo HTTPS con cifrado TLS 1.3.
El Usuario Cliente solo puede ver y cancelar sus propias reservas.
El Proveedor de Atracción puede cambiar horarios y cerrar fechas, pero no borrar reseñas.
El Administrador del Sistema puede gestionar reembolsos pero no visualizar los números de tarjeta de crédito.
REQUERIMIENTOS NO FUNCIONALES
El sistema debe exponer una API RESTful que utilice formato JSON para el intercambio de datos, permitiendo la interoperabilidad entre el servidor y diferentes clientes.
Todas las solicitudes a la API, excepto la búsqueda pública, deben estar protegidas mediante JWT o OAuth2.
La arquitectura del sistema debe estar documentada bajo el estándar OpenAPI, asegurando que todos los contratos de datos estén definidos.

Actor

Flujo Principal
1. El Actor selecciona la opción Gestión de Atracciones en el menú principal.
2. El sistema carga la interfaz y solicita los registros existentes (E1).
3. El Actor presiona el botón Nueva Atracción.
4. El sistema genera un UUID para la atracción y despliega el formulario multisección.
5. Sección 1 (Datos Básicos): El Actor ingresa Nombre, Descripciones, Dirección, Coordenadas y selecciona Ciudad/Subcategoría.
6. Sección 2 (Multimedia): El Actor carga URLs de fotos/videos, define el tipo y marca la imagen como principal.
7. Sección 3 (Actividades/Modalidades): El Actor presiona "Añadir Modalidad" e ingresa el Título por ejemplo Tour Vip, Duración e Inclusiones.
8. Sección 4 (Tarifario): Dentro de cada modalidad, el Actor añade las tarifas: Etiqueta, Rango de Edad y Precio (E2).
9. El sistema inicializa la puntuación en 0.00.
10. El Actor presiona el botón Guardar Todo.
11. El sistema valida la integridad: Nombre único, existencia de imagen principal y que los rangos de precios no sean los mismos (E3, E4).
12. El sistema ejecuta una transacción para guardar en las tablas: Atracciones, Multimedia, Modalidad y Tarifas (E1).
Flujo Alterno
6. El Actor no selecciona ninguna imagen como principal. El sistema marca la primera por defecto.
8. El Actor decide no agregar tarifas en este momento. El sistema permite guardar pero marca la atracción como "Inactiva" hasta que tenga precios.
11. El nombre de la atracción ya existe. El sistema solicita corregirlo.

F8.3.1 Registrar Pago
Descripción
Permite registrar la transacción financiera asociada a una reserva, validando el monto y actualizando el estado de la reserva a "Confirmada".
Actor
Cliente / Administrador / Sistema
Caso de Uso
Flujo Principal
1. El Actor selecciona una reserva en estado Pendiente.
2. El sistema recupera el monto total de la reserva y el código único.
3. Procesamiento:
A. Cliente (Pasarela): El sistema envía la solicitud de cobro a la pasarela externa y espera la respuesta de éxito (E4).
B. Administrador (Manual): El Actor marca la reserva como pagada tras recibir el dinero en efectivo o transferencia.
4. El sistema registra los datos del pago.
5. El sistema cambia el estado de la reserva  a Confirmado (E1).
6. El sistema genera el voucher digital de la atracción con el código QR para el cliente.
7. El sistema envía automáticamente el comprobante de confirmación al correo electrónico del usuario.
Flujo Alterno
3.A. La pasarela de pago rechaza la tarjeta. El sistema notifica al Cliente y mantiene la reserva en Pendiente por un tiempo limitado.
5. El sistema detecta que el tiempo de espera para el pago expiró. La reserva se cancela y los cupos se liberan automáticamente.
